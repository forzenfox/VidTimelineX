## 修改计划

### 1. 分析当前工作流

* 当前工作流会定期执行时间线更新脚本

* 如有变动，会创建 auto-deploy 分支并提交更改

* 会创建或更新 PR，但不会自动合并

### 2. 修改内容

#### 2.1 权限设置

* 确保工作流有足够的权限来合并 PR

* 目前已有 `contents: write` 和 `pull-requests: write` 权限，应足够使用

#### 2.2 添加自动合并步骤

* 在创建或更新 PR 后，添加自动合并的步骤

* 使用 GitHub CLI 的 `gh pr merge` 命令来合并 PR

* 添加条件检查，确保只有在满足条件时才合并

#### 2.3 合并条件设置

* 设置合适的合并条件，例如：

  * PR 状态检查通过

  * 确保 PR 是由工作流自动创建的

  * 确保 PR 标题符合预期格式

### 3. 具体修改

* 在现有的 "创建 Pull Request" 步骤后添加 "自动合并 PR" 步骤

* 使用 `if` 条件确保只在有变动时执行

* 使用 GitHub CLI 命令来检查 PR 状态并执行合并

* 添加适当的错误处理和日志输出

### 4. 验证

* 确保修改后的工作流语法正确

* 确保权限设置足够

* 确保合并条件设置合理

### 5. 预期效果

* 当工作流执行并创建 PR 后，如果满足合并条件，PR 将被自动合并

* 合并后，工作流将完成执行

* 如果不满足合并条件，PR 将保持开放状态，等待手动处理
